Postmodern

Postmodern is a Common Lisp library for interacting with PostgreSQL databases. Features are:

  • Efficient communication with the database server without need for foreign libraries.
  • Support for UTF-8 on Unicode-aware Lisp implementations
  • A lispy syntax for expressing SQL queries
  • Convenient support for prepared statements and stored procedures
  • Defclass-like definition of tables and associated accessor classes

The biggest differences between this library and CLSQL/CommonSQL are that postmodern has no intention of being portable across different SQL implementations (it embraces non-standard Postgres features), and approaches extensions like lispy SQL and database access objects in a quite different way. This library was written because the CLSQL approach did not really work for me, your mileage may vary.

News

18-09-2007: Version 1.03: A number of small bugfixes, and the introduction of call-with-connection and with-connection* to make dynamically-scoped connections more usable.

24-07-2007: Version 1.02: query and execute now return the number of effected rows as a second value (if the server returns any such statistics), the possibility to defer id creation has been added to deftable, and a few more bugs have been fixed.

13-06-2007: Version 1.01: Fix for a bug that caused problems when using dashes in the field names of a deftable form.

15-05-2007: Version 1.00: A few minor bugfixes and tweaks, most notably a fix that makes Postmodern work on LispWorks. Since the stream of bug reports seems to have more or less dried up, I bumped the version number up to 1, and this release should not be considered a beta version anymore.

16-03-2007: Version .24: Fixes :reconnect restarts, add varargs to query and execute. Fixes a bug that caused connections to go out of sync when the server sent warnings.

29-01-2007: Version .23: Fixes a few embarassing bugs in the previous release.

27-01-2007: Version .22. Another change to :create-table, this time to add support for constraints and a few other things that the plain-SQL version lets you specify. The approach to transactions (see with-transaction) was also changed. The old approach was actually incorrect, now it should be more robust.

23-01-2007: Version .21 released, which mostly corrects minor mistakes. The syntax of :create-table was changed: it no longer takes a nil/t value to indicate whether a column can be null, but uses the same (or db-null type) notation as deftable.

Dependencies

The library depends on usocket, md5, ieee-floats, trivial-utf-8 if you want Unicode support, and bordeaux-threads if you want thread-safe connection pools.

Postmodern itself is split into 4 different parts, some of which can be used independantly. Simple-date is a very basic implementation of date and time objects, used to support storing and retrieving time-related SQL types. Cl-postgres is the low-level library used for interfacing with a PostgreSQL server over a socket. S-SQL is used to compile s-expressions to strings of SQL code, doing as much as possible of the work at compile time. Finally, postmodern itself is the library that tries to put all these things together into a convenient programming interface.

Quickstart

This quickstart is intended to give you a feel of the way coding with postmodern works. Further details about the workings of the library can be found in the reference manual.

Assuming you have already installed it, first load & use the system:

(asdf:oos 'asdf:load-op :postmodern)
(use-package :postmodern)

If you have a PostgreSQL server running on localhost, with a database called 'testdb' on it, which is accessible for user 'foucault' with password 'surveiller', you can connect like this:

(connect-toplevel "testdb" "foucault" "surveiller" "localhost")

Which will establish a connection to be used by all code, except for that wrapped in a with-connection form, which takes the same arguments but only establishes the connection locally.

Now for a basic sanity test:

(query "select 22, 'Folie et déraison', 4.5")
;; => ((22 "Folie et déraison" 9/2))

That should work. But, this being Lisp, we really hate to use languages with so little parentheses in our programs:

(query (:select 22 "Folie et déraison" 4.5))
;; => ((22 "Folie et déraison" 9/2))

In truth, parentheses aren't the decisive reason for this alternate syntax. The nice thing is that you can feed lisp values directly into your query like this, without converting them to strings yourself. Queries like the one above, with only constants in them, get converted to strings entirely at compile time. When you add for example a variable or an expression, only the static part of the query gets built at compile time, and the dynamic parts are inserted at run time. Like this:

(defun database-powered-addition (a b)
  (query (:select (:+ a b)) :single))
(database-powered-addition 1030 204)
;; => 1234

That last argument, :single, indicates that we want the result not as a list of lists (for the result rows), but as a single value, since we know we are only selecting one value. Some other options are :rows, :row, :column, :alists, and :none. Their precise effect is documented in the reference manual.

You do not have to pull in the whole result of a query at once, you can also iterate over it with the doquery macro:

(doquery (:select 'x 'y :from 'some-imaginary-table) (x y)
  (format t "On this row, x = ~A and y = ~A.~%" x y))

Now let us define a table.

(deftable test-data ()
  ((field1 :type integer)
   (field2 :type (or string db-null))
   (field3 :type (numeric 10 3)))
   (:indices field1 field3))

(create-table 'test-data)

This will add a table named test_data (on the SQL side, from Lisp you can approach it as test-data), with one integer (32-bit signed) column, one column of type text, which may be null, and one of type numeric(10, 3). The table will have indices on field1 and field3, with the one on field1 being the primary index because it comes first in the index list.

You might notice that this deftable form looks quite a lot like a defclass. You can use it like that too:

(deftable test-data ()
  ((field1 :type integer :initarg :field1 :accessor field1)
   (field2 :type (or string db-null) :initform :null :accessor field2)
   (field3 :type (numeric 10 3) :initarg :field3 :accessor field3))
  (:indices field1 field3)
  (:class-name test-data))

Adding a :class-name option will result in the definition of an accessor class (dao is the term postmodern uses, which means either 'Database Access Object' or 'The Way'). This class has a slot for every column in the table, and has methods for storing or retrieving instances to and from the database.

(make-instance 'test-data :field1 44 :field3 99.9)
;; => #<TEST-DATA {ABD4E89}>
(save-dao *)
(field3 (get-dao 'test-data 44))
;; => 999/10

save-dao will save a dao, calling insert-dao if no rows with the same primary key exist, or update-dao if there is such a row. get-dao takes one argument for every column in the primary index of this table, and will return a dao if a column with those values exists.

The reason you get back a ratio (999/10) instead of the float you initially fed the object is because field3 is of type numeric, which is a fixed-precision type, not floating point.

To demonstrate a bit more of the S-SQL syntax, here is the query the utility function list-tables uses to get a list of the tables in a database:

(sql (:select 'relname :from 'pg-catalog.pg-class
      :inner-join 'pg-catalog.pg-namespace :on (:= 'relnamespace 'pg-namespace.oid)
      :where (:and (:= 'relkind "r")
                   (:not-in 'nspname (:set "pg_catalog" "pg_toast"))
                   (:pg-catalog.pg-table-is-visible 'pg-class.oid))))
;; => "(SELECT relname FROM pg_catalog.pg_class 
;;      INNER JOIN pg_catalog.pg_namespace ON (relnamespace = pg_namespace.oid)
;;      WHERE ((relkind = 'r') and (nspname NOT IN ('pg_catalog', 'pg_toast'))
;;             and pg_catalog.pg_table_is_visible(pg_class.oid)))"

sql is a macro that will simply compile a query, it can be useful for seeing how your queries are expanded or if you want to do something unexpected with them.

As you probably guessed, lists starting with keywords are used to express SQL commands and operators (lists starting with something else will be evaluated and then inserted into the query). Quoted symbols name columns or tables (keywords can also be used but might introduce ambiguities). The syntax supports subqueries, multiple joins, stored procedures, etc. See the S-SQL reference manual for a complete treatment.

Finally, here is an example of the use of prepared statements:

(defprepared select-things
  (:select 'field2 'field3 :from 'test-data :where (:= 'field1 '$1))
  :alists)
(select-things 44)
;; => (((:FIELD2 . :NULL) (:FIELD3 . 999/10)))

The defprepared macro creates a function that takes the same amount of arguments as there are $X placeholders in the given query. The query will only be parsed and planned once (per database connection), which can be faster, especially for complex queries.

Caveats and to-do's

Timezones

Simple-date has no concept of time zones. This means that using it is rather error-prone, and if you really need your time-keeping to be reliable and/or universal you should either not use the types it provides or think really hard about the way you handle time zones.

I was going to fix this by combining my code with stuff from local-time, but ran out of motivation. If you really need a reliable time datatype, you could consider finishing this.

Portability

The Lisp code in Postmodern is theoretically portable across implementations, and seems to work on all major ones. I am not actively testing against new releases or obscure implementations, but if you run into problems you are welcome to contact me through the mailing list, and we can try to solve them.

The library will definitely not work for PostgreSQL versions older than 7.4 (it uses a client/server protocol that was introduced then). On versions prior to 8.1, retrieving the simple-date classes is buggy, because their binary representation was changed.

Things that should be implemented

Support for foreign keys and constraints should be added to deftable. This will make code that automatically creates or alters tables quite a bit more complex: it will have to take care not to violate any contraints, which can be tricky in PostgreSQL.

It would be a neat feature if table-description would return a nicer representation of field types (Lisp type expressions). Some attempt could be made to add a function that compares a table's representation in the database with the corresponding table structure as defined by a deftable, and automatically (interactively) writes the queries needed to update the table in the database.

Functions for adding sequences, stored procedures, and views to the templates that deftable-style tables are currently stored in would make these a viable way to define and store database schemas.

Several of these issues are adressed by Ryszard Szopa's Submarine library, a more advanced table-declaration and access-object library built on top of Postmodern.
